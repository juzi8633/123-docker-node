1. What this project really is
--------------------------------------------------

This is not just a plain Node CRUD backend.
It is a small full-stack media-ingestion and library-management service for cloud-drive-based media workflows.

Current code shows five major responsibilities:
- maintain a local media library index in SQLite via Prisma
- parse share links from active frontend-supported sources (`123` / `189`) plus direct RapidTransfer-style JSON into normalized import data
- accept trusted imports directly or queue non-trusted imports for background verification / pending handling
- use Redis + BullMQ for background task dispatch and lock/caching behavior
- serve a Vue single-page admin UI from the same Fastify process

A good mental model is:
- `SeriesMain` = one movie or one show
- `SeriesEpisode` = actual stored playable/subtitle file rows
- `PendingEpisode` = staged tasks waiting for verification, retry, or manual handling
- backend = ingest/control pipeline for media data
- frontend = operator console

- uses `better-sqlite3`, `bullmq`, `ioredis`, `axios`, `got`, `undici`, `pino`

Interpretation:
- this project mixes HTTP control APIs with queue-driven background work
- it is not designed as a cleanly separated API/worker pair yet; the same runtime boot handles both

2.3 Docker shape
`docker-compose.yml`:
- app service exposes port `3000`
- mounts `./data:/app/data`
- Redis is separate container
- SQLite DB file is stored in mounted volume

Important mismatch to remember:
- compose sets `DATABASE_URL=file:/app/data/dev.db`
- Dockerfile sets `DATABASE_URL="file:/data/prod.db"`
- runtime final command uses `npx prisma db push && node src/app.js`

Interpretation:
- path handling matters
- `src/db.js` contains path-fix logic to compensate for Prisma/Node path differences

2.4 Dockerfile behavior
Multi-stage build:
- stage 1 builds frontend
- stage 2 installs production backend deps, copies backend code, copies built frontend to `/app/public`, runs `prisma generate`
- final startup runs schema push and backend start

Meaning:
- UI is compiled into static assets at image build time
- backend is responsible for serving SPA assets and APIs together

3. Data model
--------------------------------------------------

File: `prisma/schema.prisma`

3.1 `SeriesMain`
Represents one movie or one TV series.
Fields include:
- `tmdbId` as primary key
- `name`
- `year`
- `type` (`movie` or `tv`)
- `genres`
- `originalLanguage`
- `originCountry`
- `lastUpdated`

It is the public/library-level row.

Indexes indicate actual query intentions:
- filter by `type + originalLanguage`
- filter by `type + genres`
- sort/recent queries by `lastUpdated`
- optimized recent-updated index with `(lastUpdated desc, tmdbId, name, year)`

Interpretation:
- recent-update browsing is an intended hot path
- library search/list does not appear to use many heavy relational joins

3.2 `SeriesEpisode`
Represents actual file rows, not only TV episodes.
Fields include:
- auto-increment `id`
- `tmdbId`
- `season`
- `episode`
- `cleanName`
- `size`
- `etag`
- `S3KeyFlag`
- `score`
- `type` (`video` or `subtitle`)
- `createdAt`

Important interpretation:
- movie files also live here using season/episode `0/0`
- the same table stores subtitles and videos
- `score` is the version-selection mechanism

3.3 `PendingEpisode`
Represents staging tasks for items not directly trusted/imported.
Fields include:
- `etag`, `tmdbId`, `season`, `episode`, `cleanName`, `size`, `score`
- `type`
- `sourceType`
- `sourceRef`
- `retryCount`
- `taskId`
- `nextRetryAt`

Interpretation:
- this is a task-state table, not final library storage
- queue lifecycle and admin UI both rely on it

3.4 `SystemConfig`
Simple key-value config table.
Stores runtime credentials and operator settings such as:
- `tmdb_key`
- `account_vip`
- `account_workers`
- `cloud189_token`

Important note:
- older versions/config shapes included fields such as `quark_cookie`, but current active frontend config flow in this workspace has already been narrowed away from Quark

6. Redis and queue model
--------------------------------------------------

Files:
- `src/redis.js`
- `src/queue.js`

6.1 Redis client
`src/redis.js` exports:
- shared Redis client
- `REDIS_CONNECTION_CONFIG` for BullMQ reuse

Behavior:
- retry strategy with capped delay
- `maxRetriesPerRequest = null` for BullMQ compatibility

6.2 Queue architecture
`src/queue.js` creates:
- `downloadQueue`
- one BullMQ `Worker`

Current worker config:
- `concurrency: 1`
- limiter `max: 1, duration: 3000`

Interpretation:
- queue is intentionally serialized / throttled
- throughput is intentionally low to protect upstream pan APIs and avoid repeated races

6.3 Task behavior
Each job roughly does:
1. resolve pending row id if needed
2. acquire Redis lock by `etag`
3. dedupe against final `SeriesEpisode`
4. try rapid-upload / reuse probe via `core123.probeFileByHash(...)`
5. if reusable, write final `SeriesEpisode` row directly
6. otherwise keep the item in pending/manual-review style state instead of submitting a 123 offline-download task
7. update pending-row retry/task state as needed for the reduced verification pipeline

Important semantics:
- duplicate / lower-score files are deleted from pending queue and skipped
- subtitle/video distinction affects dedupe and replacement behavior
- pending rows are the queue truth source
- current queue path should be read as probe/verify-oriented, not offline-download-oriented

6.4 189 special path
- 189-specific task handling still exists in current backend flow where provider-specific reference resolution is needed
- token is read from Redis key `auth:189:token`

Interpretation:
- 189 support is partially normalized into the same queue pipeline
- but it still has provider-specific URL/signing or reference-resolution logic in queue worker

8. Core 123 service
--------------------------------------------------

File: `src/services/core123.js`

This is the central backend service layer for 123-pan integration.
It owns:
- VIP client
- worker client pool
- token persistence
- direct link caching
- probe/reuse logic
- cache-directory lifecycle
- direct/provider-side helper calls that still belong to the active non-offline path

8.1 Client roles
- `vipClient`: used for privileged storage/download/direct-link style operations in the active path
- `workerClients`: used for probing / upload_request reuse checks

Interpretation:
- account separation is intentional
- worker accounts absorb probe pressure
- VIP account handles final privileged operations

8.2 Token persistence
- refreshed tokens are written back to Redis under `123:token:${passport}`
- config reload reconstructs clients using DB config + cached Redis token

Interpretation:
- token storage is ephemeral-but-persistent-across-process-restarts as long as Redis survives

8.3 Direct-link fetch path
`getDownloadUrlByHash(...)`:
- first checks Redis by `123:dlink:${etag}`
- then checks in-memory inflight lock by `etag`
- if no `S3KeyFlag`, probes the file via worker account
- asynchronously backfills `S3KeyFlag` into DB
- uses VIP client to get final download URL
- caches final URL for 6 days

Important semantics:
- `etag` is the primary identity for direct-link cache and inflight de-duplication
- DB writeback of `S3KeyFlag` is intentionally async and non-blocking

8.4 Probe path
`probeFileByHash(...)`:
- uses worker account round-robin
- sanitizes filename but actually sends `.tempfile` in upload request metadata
- waits 1 second before probe as crude rate limiting
- returns `{ reuse: true, S3KeyFlag? }` or `false`

Interpretation:
- probe is really a remote rapid-upload eligibility test, not content inspection

8.5 Cache-directory management
- creates date-based cache folder names like `123_Node_Cache_YYYY-MM-DD`
- caches dir IDs in Redis
- background recycle tries to clean older folders

Important observation:
- actual trash call is commented out in `recycleAllCacheFolders()`
- current effective behavior is mostly Redis-key cleanup, not confirmed physical folder deletion
- do not claim old folders are truly deleted unless code is changed

8.6 Offline submission status
- older versions of this project exposed explicit 123 offline-download submission here
- current source in this workspace no longer keeps `addOfflineTask(...)` as an active method in `src/services/core123.js`
- do not assume a live offline-download submit path still exists just because cache-folder or VIP client behavior remains

10. Share-link parsers
--------------------------------------------------

Files:
- `src/services/service123.js`
- `src/services/service189.js`
- `src/services/serviceQuark.js`

These services have one common role:
- parse a share link
- recursively scan files
- normalize output into a common rapid-transfer JSON structure
- stream progress to frontend via SSE writer

10.1 Common export structure
All three generators eventually output something like:
- `scriptVersion`
- `exportVersion`
- `usesBase62EtagsInExport`
- `commonPath`
- `files[]`
- `totalFilesCount`
- `totalSize`

This means the import UI can treat them as one normalized upstream format.

10.2 123 parser
- validates share URL host/pattern
- fetches share title first
- recursively scans with paging
- pushes `{ path, etag, size, S3KeyFlag }`

10.3 189 parser
- resolves shareCode/shareId
- optional password check
- signs requests via MD5 over sorted params
- recursively scans folders/files
- stores `source_ref` as `fileId|shareId`

10.4 Quark parser
- obtains `stoken`
- recursively scans all files
- then batch-fetches MD5 for found files
- uses retry wrapper for unstable endpoints

Important interpretation:
- Quark integration is more expensive because MD5 is a second phase
- 189 integration preserves provider-specific reference IDs for later queue resolution
- SSE progress events are part of the API contract for frontend UX

11. Utility logic and scoring model
--------------------------------------------------

Files:
- `src/utils.js`
- `ui/src/utils/logic.js`

There are two related utility layers:
- backend utility set in `src/utils.js`
- frontend parsing/grouping/rebuild utility set in `ui/src/utils/logic.js`

11.1 Shared philosophy
Both sides encode the same broad product assumptions:
- file names are messy and need aggressive heuristics
- TMDB tagging in folder/file names is first-class
- season/episode parsing must tolerate multiple formats
- quality score decides which video version wins
- subtitles are attached to chosen best video version

11.2 Scoring model
Key current rules:
- files > 30GB => score 0
- DV / Dolby Vision => score 0
- resolution, source, codec, HDR, audio, and golden-size zone add points
- movie and TV use different preferred size ranges

Interpretation:
- this project is not neutral about file versions
- it encodes a strong opinion: online-stream-friendly and compatibility-friendly versions are preferred over archival heavy files

11.3 Frontend grouping and rebuild
`autoGroupFiles(...)`:
- filters unsupported extensions and junk extras
- extracts year/TMDB/title/season/episode info heuristically
- groups into movie/show candidates

`rebuildJsonWithTmdb(...)`:
- rebuilds normalized import JSON using TMDB-confirmed title/year/id
- chooses best video per episode/movie by score
- attaches subtitles to chosen base name

Interpretation:
- frontend does substantial pre-ingestion normalization, not just display
- backend still revalidates/standardizes later

Current recognition-chain details that matter after recent source changes:
- `parseEpisode(text)` is no longer the only episode API; `parseEpisodeDetail(text)` now exists and returns `{ value, confidence, source }`
- current confidence model is intentionally minimal: `strong | weak | none`
- strong episode signals include explicit season/episode or episode markers (for example `SxxExx`, `EPxx`, `第xx集/话`, bracketed/delimited episode patterns already present in source)
- weak episode signal currently means the loose leading-number fallback
- `extractFileInfo(fullPath)` no longer treats any episode number equally: weak episode signals do not independently force `isTV = true`; they only help when combined with season evidence
- `autoGroupFiles(rawFilesData)` now also uses episode confidence when generating TV preview episode strings:
  - strong episode signals can directly produce `_previewEpStr`
  - weak episode signals only produce `_previewEpStr` when season information is also present
- `extractFileInfo(fullPath)` now returns `fallbackSearchQuery` and `searchQuerySource` in addition to `searchQuery`; this is a minimal title-chain enhancement, not a full candidate system rewrite

12.5 Admin destructive API
- `POST /api/admin/clear-db`
  - deletes all `SeriesEpisode` and `SeriesMain`

12.6 Import/submit API
- `POST /api/submit`

This is a key route.
It supports two modes:
- trusted source (`123` or `json`) => directly writes final `SeriesEpisode`
- untrusted source (`189` and other non-trusted provider-shaped imports) => creates `PendingEpisode` rows and queues tasks

Important semantics:
- the older `handleMetadataMigration(...)` path is no longer part of the active `src/app.js` flow in this workspace
- trusted import deduplicates by `(tmdbId, season, episode)` and overwrites non-subtitle video rows in batch
- untrusted import creates staged rows first, then queues background jobs after transaction

12.7 Delete APIs
- `DELETE /api/delete/series`
- `DELETE /api/delete/episode`

12.8 Webhook upload API
- `POST /api/webhook/upload`

Behavior:
- requires `file_name` and `etag`
- extracts `{tmdb=xxx}` from file name or path
- lazily creates `SeriesMain` from TMDB if missing
- standardizes name
- computes score for videos
- skips lower-score overwrite
- writes file row and queues verification job

Interpretation:
- webhook path is an opportunistic ingestion path for newly uploaded files
- it uses score-based replacement, not version coexistence for videos

12.9 Streaming parse API
- `GET /api/stream`

This is SSE-based and delegates to the provider parser service.

12.10 123 offline callback status
- older versions exposed `POST /api/callback/123`
- current `src/app.js` in this workspace no longer keeps that route active

Important observation:
- do not read the current backend flow through the old callback-based offline-download mental model
- pending-task completion/failure handling should now be understood from the reduced queue/probe/manual-review path actually present in source

13.2 Login flow
`LoginView.vue`:

13.3 Config panel
`ConfigPanel.vue`:
- edits backend config keys
- persists via `POST ./api/config`
- after save re-fetches config

13.4 Library view
`LibraryView.vue`:
- uses `/api/search` for paged library browsing
- uses `/api/details` for expandable detail row
- supports delete series / delete episode
- supports copying series data as structured JSON

Important frontend/backend mismatch to remember:
- frontend sorts by `last_updated`, but backend response schema uses `lastUpdated`
- if sorting issues appear, inspect field naming carefully rather than assuming logic is correct

13.5 Import view
`ImportView.vue`:
- accepts raw JSON or share URLs
- detects provider type
- opens SSE parse stream via `/api/stream`
- groups parsed files locally
- batch auto-matches TMDB
- rebuilds normalized import JSON
- later submits via `/api/submit`

Interpretation:
- frontend performs a lot of ingestion intelligence before the backend write step

Current matching behavior as actually modified in source:
- automatic TMDB matching still uses item-level `isTV`, `tmdbId`, `searchQuery`, and `year`
- `autoMatchSingleItem(...)` now has a small internal helper for “search once with one query” and reuses it for the primary query
- if the primary query returns no candidate or only a low-confidence candidate (`score < 60`), it may try exactly one more search using `fallbackSearchQuery`
- this fallback does not change scoring rules or concurrency; it only adds a second query attempt when the first query is weak
- final acceptance now scores the chosen candidate against the actual query that produced it, not always the original primary query

13.6 Verification view
`VerificationView.vue`:
- polls `/api/pending/list`
- maintains selection state
- can retry selected tasks through `/api/do/`
- can bulk-delete pending rows
- current visible task semantics are aligned to `pending / processing / failed`, not the older offline-download wording

16. Important implementation truths and easy-to-misread semantics
--------------------------------------------------

16.1 Final library rows vs pending rows are distinct
- `SeriesEpisode` = accepted/finalized library file rows
- `PendingEpisode` = staged work items
Do not merge these concepts when modifying queue logic.

16.2 Trusted vs untrusted import source is a core branch
- `123` and `json` are treated as trusted in `/api/submit`
- others go through pending queue
This distinction affects whether files enter library immediately or not.

16.3 Score decides overwrite, not coexistence
For videos of same `(tmdbId, season, episode)`:
- lower-score version is skipped
- higher-score version can replace previous non-subtitle rows
Subtitles are additive alongside chosen video.

16.4 Movies are stored as episode-like rows
Movie imports use season/episode `0/0` logic in multiple places.
Do not assume `SeriesEpisode` means “TV only”.

16.5 Queue worker is intentionally serialized
The queue is not slow by accident.
Current `concurrency: 1` and limiter indicate deliberate throttling.
Do not casually raise concurrency without checking upstream API sensitivity and lock semantics.

16.6 `recycleAllCacheFolders()` does not currently guarantee remote deletion
The trash call is commented out.
Current code mainly cleans Redis directory-id cache entries.

16.7 Frontend source contains substantial business logic
Especially in import/grouping/rebuild flows.
Do not treat frontend as dumb presentation-only code.

16.8 Some naming mismatches likely exist
Examples visible from source:
- frontend library sort references `last_updated` while backend returns `lastUpdated`
- pending-task/retry payload normalization has historically crossed camelCase and snake_case boundaries
Treat such mismatches as real potential bug surfaces.

16. Safe editing guidelines for this repo
--------------------------------------------------

When modifying this repository:
- read the real route + queue + utility code before changing ingestion behavior
- distinguish trusted direct import from staged pending import
- preserve score-based best-version semantics unless intentionally redesigning it
- remember that the same process runs API + queue worker
- be careful with provider-specific fields like `S3KeyFlag`, `source_ref`, `taskId`, `cloud189_token`
- do not assume 123 / 189 / Quark share parsers are interchangeable internally; only their output format is normalized
- do not claim webhook or historical callback-style paths fully finalize ingestion unless code explicitly writes final library rows

Low-risk changes usually look like:
- field-name consistency fixes
- logging noise reduction
- better null/undefined handling in config and queue payloads
- UI/backend contract alignment
- local batching where business semantics do not change

Higher-risk changes usually look like:
- changing queue concurrency/limiting
- altering score rules
- merging pending/final tables conceptually
- changing direct-import trust semantics
- rewriting share parser recursion behavior
